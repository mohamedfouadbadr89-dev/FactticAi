Phase 3B – Hard Alert Enforcement Layer

1. Executive Summary

The Facttic Hard Alert Enforcement Layer is designed to ensure that institutional governance remains deterministic, actionable, and resilient against alert fatigue. In high-scale AI deployments, signal noise can obscure critical failures. Deterministic alert enforcement is required to gate human attention towards statistically significant breaches only.

Enforcement Objectives:

  • Eliminate volatility-induced alert noise.
  • Enforce deterministic response windows via mandatory cooldowns.
  • Provide an immutable forensic log for every severity escalation.

2. Escalation Matrix

Trigger Metric Threshold Assigned Severity
Drift Frequency Index (DFI) > 0.3 HIGH
Drift Frequency Index (DFI) > 0.5 CRITICAL
Risk Momentum Signal (RMS) > 0.4 HIGH
Governance Health Score (GHS) < 50 CRITICAL
Governance Health Score (GHS) < 40 SYSTEMIC_FAILURE

3. Cooldown Enforcement Model

To prevent signal flooding during remediation, the system enforces a deterministic cooldown period where no new alerts of the same type can be triggered for the specific organization.

Severity Level Lock Duration (Cooldown)
Elevated 6 Hours
High 12 Hours
Critical / Systemic 24 Hours

4. Duplicate Prevention Logic

The enforcement layer rejects the creation of any alert where an ACTIVE alert already exists for the same org_id and triggered_by key. This is enforced via unique partial indices and application-level pre-checks during the escalation phase.

5. Escalation Forensics Layer

Every severity state change is recorded in the governance_escalation_log for post-incident audit and SOC2 Type II compliance.

Field Description
org_id Organization Reference
metric_snapshot_id Reference to the specific governance_predictions snapshot
previous_severity State prior to escalation
new_severity State after enforcement
escalation_reason Mathematical rationale (e.g., "DFI Breach > 0.3")

6. Security Hardening

  • SECURITY DEFINER: All enforcement RPCs execute with elevated privileges to ensure RLS-protected log insertion.
  • Search Path Locking: Mandatory set_config('search_path', 'public', true) to prevent search-path injection attacks.
  • RLS Enforcement: Visibility of alerts and logs is strictly bound to organization membership.

7. Institutional Certification Criteria

Phase 3B qualifies as LOCKED only when:

  1. Unique constraints prevent duplicate active alerts.
  2. Cooldown periods are mathematically verified via database timestamps.
  3. Immuntable forensic logs reflect 100% of severity transitions.
